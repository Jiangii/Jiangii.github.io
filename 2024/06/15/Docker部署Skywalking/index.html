发布于 2024.06.15

SkyWalking 三个核⼼部分

**Skywolking Agent：**Agent 运⾏在各个服务实例中，负责采集服务实例的 Trace 、Metrics 等数据，然后通过 gRPC ⽅式上报给 SkyWalking 后端。
**Skywolking OAP：**SkyWalking 的后端服务，⼀个是负责接收 Agent 上报上来的 Trace、Metrics 等数据，进⾏流式分析，最终将分析得到的结果写⼊持久化存储中，另⼀个是负责响应 SkyWalking UI 界⾯发送来的查询请求。
**Skywolking UI：**展示查询结果。

整个架构，分成上、下、左、右四部分

  • 1）上部分 Agent ：负责从应用中，收集链路信息，发送给 SkyWalking OAP 服务器。目前支持 SkyWalking、Zikpin、Jaeger 等提供的 Tracing 数据信息。Agent 收集 SkyWalking Tracing 数据，传递给服务器。
  • 2）下部分 SkyWalking OAP ：负责接收 Agent 发送的 Tracing 数据信息，然后进行分析(Analysis Core) ，存储到外部存储器( Storage )，最终提供查询( Query )功能。
  • 3）右部分 Storage ：Tracing 数据存储。目前支持 ES、MySQL、Sharding Sphere、TiDB、H2 多种存储器。而我们目前采用的是 ES ，主要考虑是 SkyWalking 开发团队自己的生产环境采用 ES 为主。
  • 4）左部分 SkyWalking UI ：负责提供控台，查看链路等等。

Docker启动应用

Dockerfile 文件需要与它所引用的文件或文件夹位于同一个目录（或子目录）中，或者这些文件/文件夹需要位于构建上下文（build context）中，这是因为 Docker 构建过程的工作方式决定的。
docker build -t gsc_stp_m_message:1.1 .
当你运行 docker build 命令时，Docker 会创建一个构建上下文，这通常是一个包含 Dockerfile 和所有需要被复制到镜像中的文件或文件夹的目录。Docker 客户端会发送这个上下文到 Docker 守护进程（daemon），守护进程再使用这个上下文来构建镜像。
如果你指定了一个路径作为 docker build 的参数（比如 docker build . 中的点 . 表示当前目录），Docker 就会将该目录（及其子目录）中的所有文件作为构建上下文的一部分发送到 Docker 守护进程。然后，Dockerfile 中的指令（如 COPY 和 ADD）会在这个上下文中查找文件或文件夹。
如果 Dockerfile 引用了一个不在构建上下文中的文件或文件夹，Docker 就无法找到它，因此会报错说找不到文件或文件夹。这就是为什么 Dockerfile 和它所引用的所有文件或文件夹都需要位于同一个目录或构建上下文中。

默认情况下，构建上下文是 Dockerfile 所在的目录。但可以通过 -f 标志指定 Dockerfile 的位置，并通过路径参数指定构建上下文， 只是这样会将/opt下所有文件发送，太大了，即使可以使用.dockerignore排除一些文件夹和文件也不推荐
docker build -t gsc_stp_m_message:1.1 -f /opt/gsc_smart_tourism_platform/docker/message/dockerfile /opt/
