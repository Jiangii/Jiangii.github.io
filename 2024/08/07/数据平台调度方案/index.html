一、任务调度

1.1、场景分析

传统的业务系统调度基于计划的时间完成任务，实现业务处理动作。如XXL-JOB等手段。

然而在大数据平台的调度中，多个任务单元之间往往有着强依赖关系，上游任务执行并成功，下游任务才可以执行

如索道异常订单的分析，需要依赖4个任务：

​​

以及T+h的数据任务的增量数据默认依赖

​​

所以，在大数据平台中，与常规业务系统不同，调度系统的作用分为两部分：定时调度和依赖调度

1.2、现有框架

大数据工作流的调度框架常见的有

  • Apache Oozie
  • Azkaban
  • Airflow
  • DolphinScheduler

1.2.1 Apache Oozie

优点:

  • 与Hadoop生态系统深度集成： Oozie最初设计用于Hadoop，因此对Hadoop MapReduce、Pig、Hive等组件有很好的支持。
  • 强大的依赖管理： 支持复杂的依赖关系和调度策略，适合需要复杂依赖关系的任务调度。
  • 稳定性强： 作为Apache顶级项目，Oozie在大规模生产环境中被广泛使用，稳定性高。

缺点:

  • 配置复杂： 使用XML来定义工作流和协调作业，对于新用户来说可能不太友好。
  • 灵活性和扩展性相对较低： 相比其他现代工作流引擎，Oozie在动态工作流和跨平台支持方面稍显不足。

1.2.2 Azkaban

优点:

  • 易用性较好： 提供了直观的Web UI，便于管理和监控工作流。
  • 简单灵活： 通过简单的job配置文件定义任务，易于上手。
  • 适合小到中型项目： 对于不需要复杂依赖管理的场景，Azkaban是一个轻量级的选择。

缺点:

  • 功能相对有限： 相比于Airflow和DolphinScheduler，Azkaban的功能较为基础，对于复杂工作流的处理能力有限。
  • 配置复杂: 任务必须通过打一个zip包来进行实现

1.2.3 Airflow

优点:

  • 高度可编程性和灵活性： 使用Python编写工作流，对于数据工程师非常友好，可以轻松实现复杂逻辑。
  • 活跃的社区支持： Airflow拥有庞大的用户群和活跃的开源社区，持续贡献新插件和特性。
  • 丰富的连接器和操作符： 支持大量数据源和服务的连接，易于与其他系统集成。

缺点:

  • 资源消耗较高： 在大规模部署时，由于其基于Celery或Kubernetes的执行模式，可能会消耗较多计算和存储资源。
  • 运维复杂度： 高度的灵活性也意味着运维复杂度相对较高，特别是在规模扩大时。

1.2.4 DolphinScheduler

优点:

  • 图形化界面： 提供了直观的Web UI进行工作流设计和管理，降低了使用门槛。
  • 高性能和高可用性： 设计上注重性能优化，支持分布式部署，保证系统的稳定性和高可用性。
  • 支持多种任务类型： 不仅支持大数据生态中的任务，还涵盖了Shell、SQL等多种任务类型，适用范围广。

缺点:

  • 相对年轻： 相比Oozie和Airflow，DolphinScheduler是一个较新的项目，社区和生态还在发展中。
  • 国际化程度： 虽然在中文社区中有较好的影响力，但国际市场的文档和社区支持可能不如其他几个项目成熟。

显然，我们只需要对Airflow 和 DolphinScheduler 做对比，选择合适的方案。

1.3、方案选择

简而言之，airflow是使用Python编写工作流，DolphinScheduler 开放的api是 java的，我们是用java，所以选择DolphinScheduler 。

二、资源调度

2.1、场景分析

资源调度主要负责管理集群的计算资源（CPU、内存等），确保这些资源能够合理、高效地分配给需要执行的任务。它关注的是物理资源的分配和回收，以达到资源的最大化利用。

一个集群调度工具有多个目的：使得集群的资源被高效利用，支持用户提供的配置约束（placement constraint），能够迅速调度应用以此保证它们不会处于待定状态（pending state），有一定程度的“公平”（fairness），具有一定的鲁棒性和可用性

2.2 现有框架

2.2.1Apache YARN（Yet Another Resource Negotiator）:

  • 特点: YARN 是Hadoop生态系统的核心组件之一，专为大规模数据处理设计，提供了通用的资源管理框架。它支持多租户、高可用性，并能够有效管理集群资源，同时支持多种计算框架（如MapReduce, Spark, Flink等）。
  • 适用场景: 适用于已有的或计划构建基于Hadoop生态的大数据处理环境，尤其是当需要支持多种计算模型时。

2.2.2Kubernetes (K8s):

  • 特点: K8s 是一个开源的容器编排平台，起初为Docker容器设计，但现在已经支持其他容器运行时。它提供了强大的服务发现、负载均衡、自动扩缩容等功能，特别适合微服务架构和云原生应用。
  • 适用场景: 当企业已经采用或计划全面拥抱容器化和云原生技术栈时，Kubernetes是一个极佳的选择。它能很好地与大数据处理框架如Spark、Flink的容器化版本集成，提供高度灵活和可扩展的资源调度能力。

2.2.3 Apache Mesos:

  • 特点: Mesos 是一个分布式系统内核，提供有效的资源隔离和共享，支持多种工作负载类型。它通过两级调度架构，允许运行多个框架，如Marathon（长服务）、Chronos（定时任务）以及Mesos Native Docker集成。
  • 适用场景: 当组织需要在同一个集群上运行不同类型的作业，且这些作业具有多样化的资源需求时，Mesos是一个很好的选项。

2.2.4 Apache Hadoop MapReduce v1:

  • 特点: 虽然较旧，但在某些特定场景下仍有其应用价值，特别是对于那些已经稳定运行且不需要YARN提供的灵活性和高级功能的系统。
  • 适用场景: 对于维护成本较低、对新技术接纳度不高的成熟系统，或者对于简单的批处理作业来说，MapReduce v1可能仍然是一个可行的选择。

2.3方案选择

首先排除MapReduce 。没有价值

从大数据生态来说，yarn相对成熟，k8s则是容器编排领域更强大的

2.3.1 mesos 和yarn

优先对比Mesos和yarn，其实2者差别不大，都可以使用。

在Mesos中，各种计算框架是完全融入Mesos中的，也就是说，如果你想在Mesos中添加一个新的计算框架，首先需要在Mesos中部署一套该框架；而在YARN中，各种框架作为client端的library使用，仅仅是你编写的程序的一个库，不需要事先部署一套该框架。从这点上说，YARN运行和使用起来更加方便。所以，优先选用yarn。

2.3.2 yarn跟 k8s

Kubernetes 和 YARN 各自具有独特的优势和不足。Kubernetes 的优势在于其灵活性、可扩展性和可靠性。它提供了丰富的功能，如自动部署、自动扩缩容、服务发现等，并支持多种容器运行时。此外，Kubernetes 还具有强大的社区支持和生态系统，使得用户可以轻松找到和集成各种工具和服务。

然而，Kubernetes 的复杂性可能会成为其劣势，尤其是在入门和学习曲线方面。它需要一定的配置和管理，并且对于初学者来说可能具有挑战性。此外，Kubernetes 的资源占用和性能开销也可能成为一些小型项目的瓶颈。

相比之下，YARN 的优势在于其高性能和低延迟的大数据处理能力。它能够处理大规模数据集，并提供了各种数据处理框架的支持，如 Hadoop MapReduce 、Spark 和 Flink 等。此外，YARN 还具有简单易用的 API 和工具集，使得开发人员可以快速构建和管理大数据应用程序。

然而，YARN 的局限性在于它主要关注大数据处理领域，与其他工具的集成可能不够灵活。此外，YARN 的设计决策和生态系统相对较小，与 Kubernetes 相比可能没有那么丰富的选择和生态系统。

​​

但我们需要的是专注于高性能大数据处理的资源管理系统，YARN 是更好的选择。它提供了高效的大数据处理能力，适用于各种用例，并具有简单易用的 API 和工具集。

其实目前有混合部署的方案，如小红书，采用了开源的Koordinator

小红书是 Koordinator 社区的活跃成员，为了进一步丰富 Koordinator 支持的在离线混部场景，社区会同来自阿里云、小红书、蚂蚁金服 的开发者们共同启动了 Hadoop YARN 与 K8s 混部项目，支持将超卖的 Batch 资源提供给 Hadoop YARN 使用，进一步提升集群资源的使用效率，该项目目前已经在小红书生产环境正式投入使用。

​​

ResourceManager 和 NodeManger 是 YARN 的核心组件，ResourceManager 在管控侧负责接收任务以及资源调度，NodeManager 负责任务的生命周期管理。在 YARN & K8s 混部场景下，RM 将仍然作为 YARN 集群的核心组件独立部署，NM 将以容器的形式部署。

Koordinator 新增了 koord-yarn-operator 模块，负责将 Batch 资源量同步给 YARN RM。为了对资源进行更精细的管理，YARN task 将与 NM 的资源管理相互独立，NM 在部署时只需按自身开销申请 Batch 混部资源。YARN 任务的资源使用通过 cgroup 来管理（LinuxContainerExecutor 模式），将 cgroup 路径在 besteffort Pod QoS 下，确保可以和其他 K8s Pod 一样，统一在 besteffort 分组下管理。

koodlet 目前在单机支持了一系列的 QoS 策略，这些同样需要针对 YARN 场景进行适配。对于资源隔离参数，例如 Group Identity，Memory QoS，L3 Cache 隔离等，koordlet 将根据设计的 cgroup 层级进行适配。而对于驱逐和压制这类动态策略，koordlet 将新增一个 sidecar 模块 koord-yarn-copilot，用于对接 YARN 场景的各类数据和操作，包括 YARN task 元信息采集、资源指标采集、task 驱逐操作等，所有 QoS 策略仍然保留在 koordlet 内，koordlet 内部相关模块将以 plugin 形式对接 koord-yarn-copilot 接口。同时，koord-yarn-copilot 的接口设计将保留一定的扩展性，后续可用于对接其他资源框架。

2.4 方案概述

​​

Yarn包括两个部分：一个是资源管理器（Resource Manager），一个是节点管理器 （Node Manager）。这也是Yarn的两种主要进程：ResourceManager进程负责整个集群的资源调度管理，通常部署在独立的服务器上；NodeManager进程负责具体服务器上的资源和任务管理，在集群的每一台计算服务器上都会启动，基本上跟HDFS的DataNode进程一起出现

工作流程：

1.我们向Yarn提交应用程序 ，包括MapReduce ApplicationMaster、我们的MapReduce程序，以及MapReduce Application启动命令。

2.ResourceManager进程和NodeManager进程通信 ，根据集群资源，为用户程序分配第一个容器，并将MapReduce ApplicationMaster分发到这个容器上面，并在容器里面启动MapReduce ApplicationMaster。

3.MapReduce ApplicationMaster启动后立即向ResourceManager进程注册，并为自己的应用程序申请容器资源。

4.MapReduce ApplicationMaster申请到需要的容器后，立即和相应的NodeManager进程通信，将用户MapReduce程序分发到NodeManager进程所在服务器，并在容器中运行，运行的就是Map或者Reduce任务。

5.Map或者Reduce任务在运行期和MapReduce ApplicationMaster通信，汇报自己的运行状态，如果运行结束，MapReduce ApplicationMaster向ResourceManager进程注销并释放所有的容器资源。

MapReduce如果想在Yarn上运行，就需要开发遵循Yarn规范的MapReduce ApplicationMaster，相应地，其他大数据计算框架 也可以开发遵循Yarn规范的ApplicationMaster，这样在一个Yarn集群中就可以同时并发执行各种不同的大数据计算框架，实现资源的统一调度管理
